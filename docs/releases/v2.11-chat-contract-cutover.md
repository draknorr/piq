# v2.11 - Chat Contract Cutover and TigerData Operating Model

**Release Date:** April 6, 2026

Version 2.11 reflects the current Tiger/query-api cutover state. The product experience remains centered on PublisherIQ's normal dashboard surfaces, but the chat and search/discovery runtime now runs through a Tiger-backed contract service instead of the earlier migration scaffolding.

## Highlights

### Chat and Search

- `/chat` now routes supported answers through `query-api` and Tiger-backed contracts.
- Chat continuation, entity resolution, semantic retrieval, catalog search, momentum discovery, change explanation, and recent-news search all use the typed contract layer.
- The chat stream now carries richer end-of-turn metadata for quality, session carry-forward, execution trace, and Tiger primary/shadow diagnostics.

### Operational Model

- `query-api` is the contract boundary between the Vercel app and TigerData.
- TigerData is now the serving backend for the live contract layer, while Supabase remains the write/control plane and still serves the main non-chat product surfaces.
- Preview and production now use separate Tiger/query-api deployment pairs, and deployed Vercel environments must set `QUERY_API_BASE_URL` explicitly.

### Readiness and Coverage

- `searchDocuments` is now ready after exact parity for the docs/news slice was validated.
- The contract registry now includes the ready Tiger surface for entity lookup, overview, related-entity expansion, ranking, comparison, history tracing, change explanation, change discovery, and user-context retrieval.
- Tiger production refresh automation now keeps the serving slices synchronized through dedicated preview and production sync workflows.

## What This Means For Operators

- Use the Tiger/query-api docs for the current serving model and deployment topology.
- Use the historical Tiger specs only as dated decision records and parity evidence.
- Keep user-facing docs focused on product behavior rather than infrastructure branding.
